## Comparing SFT and RL

**Supervised Fine-Tuning (SFT)** fits best when target behaviors come as demonstrations (instruction-response pairs). It is effective for teaching format and style, but it has three limitations:

- **Imitation over adaptivity**: With a small dataset, models mimic the answer rather than the process to reach it.
- **Data bottlenecks**: Curating high-quality demonstrations or large datasets for every edge case is expensive.
- **Brittleness**: SFT models struggle outside their training distribution, so the dataset needs breadth and volume.

**Reinforcement Learning (RL)** is the better choice as complexity grows. Instead of telling the model "say exactly this," RL provides a goal and a way to verify it. The model explores reasoning paths and becomes resilient to edge cases. RL works well for math, code, and tool calling — tasks with a clear path to answer verification.

### Combining SFT + RL

SFT and RL are not mutually exclusive. A hybrid strategy is common:

- **SFT for warm-starting RL**: Use high-quality demonstrations to teach the chat template, tool-calling format, and general readability. RL then doesn't waste cycles learning dataset format.
- **RL for scaling**: Transition to RL so the model explores and self-corrects. This post-training refinement is where reasoning and robustness emerge.

The [NVIDIA Nemotron 3](https://research.nvidia.com/labs/nemotron/Nemotron-3/) family uses SFT as a substantial first stage to ground the model before RL refinement. The right balance depends on compute budget, data availability, and the level of generalization the agent requires. The industry is shifting more compute toward RL stages, especially as RL environments become more sophisticated and accessible.

## From Algorithms to Environments

PPO (Proximal Policy Optimization) was the traditional RL method. Its resource intensiveness — multiple complex models including reward and critic models — drove a shift toward more scalable algorithms.

Modern workflows adopt more efficient methods for different aspects of model improvement.

### DPO

**Direct Preference Optimization (DPO)** sidesteps the RL loop entirely and treats alignment as a classification problem on static preference data.

- **Reward type**: Pairwise. Relies on labeled preferences ("Response A > Response B").
- **Efficiency**: Computationally light and stable, ideal for alignment tasks (safety, tone, style).

DPO lacks explicit reward optimization or exploration. It learns from fixed preference pairs, so it cannot discover new strategies or optimize long-horizon outcomes. Because DPO models relative output preference rather than trajectory reward, it is less effective for agentic workflows requiring multi-step reasoning and tool use.

### GRPO

Algorithms that leverage verifiable rewards address these limitations in agentic domains.

**Group Relative Policy Optimization (GRPO)** is an optimized version of PPO that replaces heavy critic models by generating groups of outputs and scoring them against a deterministic verifier.

- **Reward type**: Typically binary (0 or 1), but supports continuous values. GRPO thrives when an environment can programmatically say "Yes" or "No" (for example, passing a unit test) and also supports more granular scoring.
- **Efficiency**: Eliminating the value model and the reward model from PPO significantly reduces memory overhead and is a key factor in scaling reasoning capabilities.

During GRPO training, **rollouts** come from interacting with an environment: complete attempts at a task, from initial prompt through tool usage to a final reward score. The environment scores each rollout in a group, and the algorithm uses those scores to update the model in real time.

### The Rise of RLVR

The shift toward verifiable correctness is distinct from any single algorithm. Verification drives improvements even in supervised settings (such as rejection sampling), but it is central to the paradigm of **Reinforcement Learning from Verifiable Rewards (RLVR)**.

RLVR replaces subjective scoring with explicit checks: *Did the agent produce the correct answer? Did it call the right tools?* This moves the "center of gravity" from the optimizer to the environment. Algorithms like GRPO provide an efficient mechanism to optimize against these environmental signals.
