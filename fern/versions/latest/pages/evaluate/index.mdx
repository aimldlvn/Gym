Run a benchmark, compare agents and models, score against your own data, profile the rewards. This section is for users whose primary goal is evaluation, not training. Eval-native vocabulary is the surface here — pass@1, accuracy, score — and the underlying training plumbing surfaces only when it's load-bearing.

## The Mental Model — `gym eval`

The conceptual one-liner is straightforward: pick a benchmark, an agent, and a model, and you get a score. Today, that flow is split across three commands:

```bash
1. ng_run +config_paths=[...]                # start servers
2. ng_collect_rollouts +agent_name=... ...   # collect responses
3. ng_reward_profile +input_jsonl_fpath=...  # compute pass rates
```

Future versions may consolidate these into a single `gym eval` invocation. For now, the [Run a Benchmark](/latest/evaluate/run/run-a-benchmark) page walks through each command end-to-end.

## What You Choose

Every eval run is the combination of four components. Pick them once, and the same flow scores any task.

- **Environment** — The benchmark or customer eval. Defines the task, the verification logic, and the dataset.
- **Agent** — The harness that orchestrates the rollout. `simple_agent` covers most evals (it runs the model + tool-call loop until the model emits a final answer); refinement loops or external runtimes use a richer agent.
- **Model** — The LLM under evaluation. Any OpenAI-compatible endpoint works — OpenAI proper, Azure, build.nvidia.com, self-hosted vLLM.
- **Sandbox** — Where code runs, when applicable. Local subprocess, Ray, or Docker, depending on the benchmark's isolation needs.
