An agent in NeMo Gym (also called the agent harness, exposed as the agent server) is the orchestrator that runs the model + tools loop and forwards model output to the resources server for verification. NeMo Gym ships several agent harnesses for common patterns. This page helps you pick one — or shows you when to bring your own.

<Tip>
If your task is single-turn or simple multi-step tool calling, start with [`simple_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/simple_agent). Pick another harness only when you have a specific reason — multi-turn refinement, an external library you already trust, or a domain-specific loop (SWE, browse, terminal).
</Tip>

## Agent Catalog

Filter by shape (model+tool loop, correction loop, external wrapper, domain-specific) or by capability (trainable multi-turn, uses Gym `verify()`). Pair any of these with a resources server and a model server in your YAML config.

## Bring Your Own Agent

Two paths, depending on whether you already have an agent runtime you trust.

### Wrap an external library

LangGraph, CrewAI, NeMo Agent Toolkit, Claude Code-style harnesses, and most other agent runtimes integrate at the agent server level. The pattern:

1. Subclass `SimpleResponsesAPIAgent`.
2. In `/run`, pre-process the Gym JSONL row into the library's input shape.
3. Drive the external loop. Use NeMo Gym's OpenAI client (`nemo_gym/openai_utils.py`) for model calls; use aiohttp for any external HTTP. Avoid `httpx.AsyncClient` — it has O(n²) connection-pool behavior at high concurrency.
4. Post-process the library's output back into a `BaseVerifyResponse`.
5. Reproduce publicly reported numbers with the original repo first, then again after Gym integration.

Refer to [Integrate External Environments](/latest/build-environments/integrations/external-libraries) for the full walkthrough; [`verifiers_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/verifiers_agent) and [`langgraph_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/langgraph_agent) are reference implementations.

### Implement `SimpleResponsesAPIAgent` directly

Subclass [`SimpleResponsesAPIAgent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents) and implement `responses()` and `run()`. For multi-turn agents:

- Propagate cookies through every downstream call: `cookies=request.cookies`. Stateful resources servers depend on this.
- Token IDs (`prompt_token_ids`, `generation_token_ids`, `generation_log_probs`) ride along on each output item the model server returns. The harness propagates them by re-sending the full prior `output` list as the next turn's input — don't drop or rebuild output items, since NeMo RL training reads those fields off them. (Note: `proof_refinement_agent` deliberately resets context per correction turn, so it is not directly trainable as a multi-turn trajectory.)
- Handle tool failures and bad model outputs as meaningful error responses, not crashes — `/run` must survive 4k–65k concurrent requests.

[`proof_refinement_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/proof_refinement_agent) is a clean reference for a multi-turn correction loop. For state machines, copy [`gymnasium_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/gymnasium_agent).

## Gym Agents vs. External Frameworks

When users ask "should I use `simple_agent` or LangGraph / CrewAI / Agent Toolkit?", the answer depends on what NeMo Gym is providing.

| You want… | Use a Gym agent | Use the external framework directly |
|---|---|---|
| Reward verification (`verify()` per task) | Yes | No |
| Rollout collection at scale (`ng_collect_rollouts`, 4k–65k concurrent) | Yes | No |
| Per-task `pass@k` profiling for RL training data (`ng_reward_profile`) | Yes | No |
| Just an agent runtime for a one-off app | — | Yes |
| Production deployment of a single agent | — | Yes |
| Already have a LangGraph / Verifiers / Harbor flow you want to evaluate | Wrap it in a Gym agent | — |

If you need Gym's verification + rollout collection, use a Gym agent (or wrap your external one). If you just need an agent runtime, the external framework is the right tool.
