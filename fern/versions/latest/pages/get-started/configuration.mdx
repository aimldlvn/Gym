NeMo Gym reads model credentials from a top-level `env.yaml` file in the project root. This keeps secrets out of YAML configs and out of version control (`env.yaml` is gitignored).

<Tip>
Already have `env.yaml` from a previous run? Jump to the [Quickstart](/latest/get-started/quickstart).
</Tip>

## Do You Need an API Key?

Not always. Three modes:

| Scenario | API key needed? |
| --- | --- |
| Calling **OpenAI**, **Azure OpenAI**, or any auth'd endpoint | ✅ Yes — real key, validated by the provider |
| Calling **self-hosted vLLM** or any local OpenAI-compatible server | ⚠️ The `policy_api_key` field must exist in `env.yaml`, but the value is ignored by the local server. Use any string (e.g. `not-needed`) |
| Running metadata-only CLIs: `ng_help`, `ng_list_benchmarks`, `ng_dump_config`, `ng_prepare_data` | ❌ Not needed at all — these commands skip the model config entirely |

If you're only listing benchmarks or preparing data, you can skip the rest of this page.

## Prerequisites

1. NeMo Gym installed using the [Installation Guide](/latest/get-started/installation).
2. An OpenAI-compatible inference endpoint and API key (refer to [Prerequisites — API and Service Access](/latest/get-started/prerequisites#api-and-service-access)).
3. Virtual environment activated.

## Configure Your Model Credentials

Create `env.yaml` in the project root with three required fields:

| Field | Description |
| --- | --- |
| `policy_base_url` | OpenAI-compatible endpoint URL |
| `policy_api_key` | Your API key for that endpoint |
| `policy_model_name` | The [policy model](/latest/resources/glossary#policy-model) to call (the LLM being evaluated or trained) |

```bash
cat > env.yaml << 'EOF'
policy_base_url: https://api.openai.com/v1
policy_api_key: sk-your-actual-key-here
policy_model_name: gpt-4.1-2025-04-14
EOF
```

Replace the placeholder API key with your real key.

`git status env.yaml` should return no output — the file is gitignored.

<Tip>
**Why `gpt-4.1-2025-04-14` for the Quickstart?** Low latency (no reasoning step) and reliable function-calling support, so the Quickstart runs in seconds without configuration surprises. Swap in any model later.
</Tip>

## Provider Examples

Same three fields, different endpoint shape per provider.

| Provider | `policy_base_url` | `policy_api_key` | `policy_model_name` |
| --- | --- | --- | --- |
| **OpenAI** | `https://api.openai.com/v1` | `sk-...` | `gpt-4.1-2025-04-14` |
| **Azure OpenAI** | `https://<resource>.openai.azure.com/openai/v1` | Azure key | `gpt-4.1-2025-04-14` |
| **vLLM (self-hosted)** | `http://localhost:8000/v1` | `not-needed` | `Qwen/Qwen3-4B-Instruct-2507` |

For self-hosted models, function-calling support is required for tool-using agents.

## Coming From Another Framework

If you already train with one of these frameworks, here is how `env.yaml` maps to what you know.

| Your framework | Equivalent in `env.yaml` | Notes |
| --- | --- | --- |
| **NeMo RL** | `policy_base_url` points at your generation backend; `policy_model_name` matches the served model name in your NeMo RL config | NeMo Gym calls the same OpenAI-compatible endpoint your training loop uses for rollout generation |
| **veRL** | `policy_base_url` is the inference server you serve from (vLLM, SGLang, etc.); `policy_model_name` is the served alias | Wire NeMo Gym to the same inference endpoint your training loop uses; keys/secrets stay in `env.yaml`, never in environment configs |
| **Unsloth** | `policy_base_url: http://localhost:8000/v1` (vLLM served by Unsloth); `policy_model_name` matches the LoRA adapter name | Refer to the [Unsloth tutorial](/latest/train-models/unsloth) for end-to-end wiring |

The same `env.yaml` is read by every NeMo Gym CLI — `ng_run`, `ng_collect_rollouts`, `ng_reward_profile`, and the others. You configure once.

## CLI Overrides

Override any field on the command line with `++key=value`:

```bash
ng_run "+config_paths=[...]" ++policy_model_name=gpt-4o-mini
```

Use this for one-off runs without editing `env.yaml`.

## Troubleshoot Configuration

<Accordion title="Missing mandatory value: policy_api_key">
`env.yaml` does not exist or is not in the project root. From the project root:

```bash
ls env.yaml
cat env.yaml | head -3
```

Re-create the file using the snippet above.
</Accordion>

<Accordion title="OpenAI 401: invalid API key">
Verify the key matches one of your active keys at [platform.openai.com/api-keys](https://platform.openai.com/api-keys). Ensure no quotes, whitespace, or newlines around the value.
</Accordion>

<Accordion title="Model not found / 404">
The model name in `policy_model_name` may not be available on your account. Try `gpt-4o-mini` or check the model list with:

```bash
python -c "from openai import OpenAI; c = OpenAI(); print([m.id for m in c.models.list().data])"
```
</Accordion>

<Accordion title="404 with double-slash in the URL">
A trailing slash on `policy_base_url` produces double-slash request paths (for example, `https://api.example.com/v1//responses`) that some providers reject as 404. Remove the trailing slash:

```yaml
# Wrong
policy_base_url: https://api.example.com/v1/

# Right
policy_base_url: https://api.example.com/v1
```
</Accordion>
