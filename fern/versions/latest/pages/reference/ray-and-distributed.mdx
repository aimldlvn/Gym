NeMo Gym uses Ray for distributed coordination across CPU-bound work in resources servers and tools. Ray initializes automatically when you start servers; you write `@ray.remote` functions and the Gym schedules them across the cluster.

## Initialization

Ray initializes in two places when `ng_run` starts:

| Location | Role |
|----------|------|
| Main process (`cli.py`) | `RunHelper.start()` calls `initialize_ray()` once. |
| Each server process (`server_utils.py`) | Workers connect to the same cluster started by the main process. |

If `ray_head_node_address` is unset, NeMo Gym starts a local cluster and writes its address to the global config so child processes can attach.

## Connect to a remote cluster

Set `ray_head_node_address` in your config to attach to an existing cluster:

```yaml
ray_head_node_address: "ray://your-cluster-address:10001"
```

Training frameworks like [NeMo RL](https://github.com/NVIDIA-NeMo/RL) typically configure this for you so remote tasks fan out across all training nodes.

## Parallelize CPU-intensive work

Decorate the expensive function with `@ray.remote` and submit calls with `.remote(...)`. Use `SPREAD` so tasks distribute across nodes rather than piling onto the head:

```python
import ray


@ray.remote(scheduling_strategy="SPREAD")
def cpu_intensive_task(data):
    return expensive_computation(data)


def process_data_parallel(data_list):
    futures = [cpu_intensive_task.remote(d) for d in data_list]
    return ray.get(futures)
```

In async code (a verifier's `verify()` or an agent's `run()`), `await` futures directly — never call `ray.get()` from inside the event loop:

```python
result = await cpu_intensive_task.remote(data)
```

For the full `@ray.remote` API, see the [Ray documentation](https://docs.ray.io/en/latest/ray-core/api/doc/ray.remote.html).
